# 微信 Claude Code 助手：产品与架构说明

## 一句话介绍

这是一个把“微信”变成本地 Claude Code 远程任务入口的工具。

你可以在手机上通过微信给本机 Claude Code 派任务，工具会在本地后台执行，并把过程状态和最终结果回发到微信。

## 它解决了什么问题

很多人使用 Claude Code 时有一个现实问题：

- Claude Code 很强，但它通常绑定在本地电脑和终端上
- 人离开电脑之后，无法方便地继续派活或查看结果
- 临时想到一个任务，往往要等回到电脑前才能开始

这个项目解决的，就是“移动端触达本地代码代理”的问题。

更具体地说，它把下面这条链路产品化了：

```text
微信消息
  -> 本机微信桥接
  -> 本地 Claude Code 执行
  -> 结果再回微信
```

这样一来，用户在通勤、会议间隙、外出途中，也可以通过微信给本地开发环境派任务。

## 这到底是一个什么产品

从产品形态看，它**不是纯 SDK**，也**不是纯 Channel 插件**。

更准确的表述是：

- 对最终用户，它是一个 **本地可安装、可后台运行的微信派活工具**
- 对工程结构，它内部采用了 **SDK 式桥接抽象**
- 对高级用户，它额外提供了一个 **可选的 Channels 适配层**

所以它的产品定位应当理解为：

> 一个 `worker-first` 的本地 AI Agent 桥接产品，内部保留 SDK 化抽象，并兼容高级 Channels 模式。

## 为什么主路径不是 Channels

因为 Channels 很强，但它不适合作为普通用户的默认安装路径。

Channels 的问题在于：

- 它依赖 Claude Code 当前会话在线
- 它依赖 Claude Code 环境本身支持 Channels
- 它更像“会话内扩展”，不是“后台派活产品”

而用户真正想要的是：

- 不懂技术也能装
- 扫码后就能用
- 不用一直守着终端
- 微信上发一句，本机就开始干活

这就是为什么本项目最终采用 `worker-first` 作为主路径。

## 产品如何工作

### 主路径：worker 模式

主路径的工作流如下：

1. 用户在微信里发任务
2. 本地 worker 通过 iLink API 长轮询拿到消息
3. worker 校验发送者是否已授权
4. worker 把任务交给 Claude Code 子进程执行
5. worker 在微信里回传阶段状态
6. worker 把最终结果发回微信

在当前版本里，这条主路径已经不只处理纯文本任务，也支持：

- 用户直接发图片
- 用户发图片再补文字
- 用户在文字里附图片链接

bridge 会尽量把图片先落到本地，再把图片路径和文字一起交给 Claude。

这是一个“后台执行型”产品，而不是“当前终端会话桥接型”产品。

### 高级路径：channels 模式

高级模式保留，但不作为默认路径。

它适合：

- 已经在 Claude Code 终端里工作的人
- 想把微信消息直接桥接进当前会话的人
- 需要研究 `claude/channel` 协议的人

## 系统架构

### 总体架构

```text
WeChat / ClawBot
        |
        v
WeChat Bridge Core
        |
        +--> worker 模式：本地后台执行 Claude Code
        |
        +--> channels 模式：桥接到当前 Claude Code 会话
```

### 核心模块

#### 1. WeChat Bridge Core

作用：

- 封装微信 iLink API
- 负责消息拉取和消息回发
- 负责把图片消息或图片链接转成可供本地 Agent 使用的输入文件
- 屏蔽协议细节
- 为不同模式提供统一桥接接口

意义：

- 把微信协议和上层产品逻辑解耦
- 后续既可以支持 worker，也可以支持 channels
- 为未来做 SDK 化提供基础

#### 2. Worker Runtime

作用：

- 轮询微信消息
- 做用户授权校验
- 管理任务生命周期
- 启动 Claude Code 子进程
- 记录状态、日志和 PID

意义：

- 这是真正的产品主干
- 负责“后台可用”“服务可控”“任务可串行”的能力

#### 3. Claude Runner

作用：

- 调用本地 `claude` 命令
- 管理会话恢复
- 控制权限模式
- 回收执行结果

意义：

- 把“Claude Code 执行”从业务逻辑里抽出来
- 更容易测试
- 更容易替换成别的 Agent 后端

#### 4. Channels Adapter

作用：

- 把微信消息转成 `claude/channel` 通知
- 把 Claude 的微信回复工具调用再桥回微信

意义：

- 提供高级交互能力
- 保留协议研究价值
- 但不干扰主路径

## 设计思路

### 1. 先可用，再高级

项目最重要的设计取舍是：

- **先把 worker 主路径做稳**
- 再把 channels 作为高级模式保留

这比“只做一个很酷的 Channel 插件”更符合真实用户价值。

### 2. 用户不需要理解协议

普通用户不应该知道这些概念：

- MCP
- Channel
- iLink
- sync buffer
- session store

普通用户应该只知道：

1. 安装
2. 扫码
3. 微信发消息
4. 收结果

现在还可以自然地理解成：

1. 发文字
2. 发图片
3. 发图片链接
4. 等结果

### 3. 让后台运行成为默认思维

这个产品真正的爽点，不是“能聊”，而是“能派活”。

所以系统设计上必须把下面这些能力做成产品级能力：

- 后台常驻
- 开机自动启动
- 同一会话串行执行
- 状态回传
- 失败回传
- 会话恢复

### 4. 主路径和高级路径必须隔离

一个关键经验是：高级模式不能污染主路径。

这就是为什么后面补了这些隔离设计：

- worker 和 channels 使用独立 sync cursor
- worker 子进程强制空 MCP 配置
- channels 只在显式命令下启用
- `doctor` 默认检查 worker，advanced 检查显式开启

## 用户怎么快速使用

### 首次安装

```bash
npm install
npm run build
node dist/cli.js install
```

安装向导会做：

1. 环境检查
2. 默认模式选择
3. 微信扫码登录
4. 工作目录与执行策略设置
5. 自动启动偏好保存

### 启动

前台运行：

```bash
node dist/cli.js start
```

后台运行：

```bash
node dist/cli.js start --daemon
```

安装开机自动启动：

```bash
node dist/cli.js service install
```

### 在微信中交互

常用命令：

- `/echo 你好`：连通性测试
- `/status`：查看当前任务状态
- `/reset`：重置当前微信会话对应的 Claude 会话
- `/help`：查看帮助

多模态输入示例：

- “请看这张截图，告诉我报错最可能的原因。”
- “请结合这张页面截图和我上面的描述，帮我判断布局为什么错位。”
- “请看这个图片链接，帮我提取图中的关键信息。”

普通任务直接发自然语言即可，例如：

```text
请检查当前项目的 README，并告诉我主路径是不是 worker 模式。
```

### 停用

停止当前后台 worker：

```bash
node dist/cli.js stop
```

如果之前启用了自动启动，再重新启用：

```bash
node dist/cli.js service install
```

如果想移除自动启动：

```bash
node dist/cli.js service uninstall
```

## 与竞品相比，它的定位和优势是什么

下面主要对比两个最接近的参考项目：

- [`wong2/weixin-agent-sdk`](https://github.com/wong2/weixin-agent-sdk)
- [`Johnixr/claude-code-wechat-channel`](https://github.com/Johnixr/claude-code-wechat-channel)

### 对比 `wong2/weixin-agent-sdk`

`weixin-agent-sdk` 的强项是：

- 抽象干净
- SDK 化程度高
- 支持 ACP
- 容易接入多种 Agent 后端

它更像一个 **底层桥接框架 / SDK**。

本项目和它的区别在于：

- 本项目不是只停留在 SDK 抽象，而是进一步做成了 **面向 Claude Code 的成品工具**
- 本项目补了更明确的本机运行路径：安装向导、后台模式、`launchd`、状态命令、停止命令
- 本项目把“微信派活到本地 Claude Code”这个单场景做了更强的产品化收口

可以理解为：

- `weixin-agent-sdk` 更像“平台能力”
- 本项目更像“围绕 Claude Code 的垂直产品”

### 对比 `Johnixr/claude-code-wechat-channel`

`Johnixr` 项目的强项是：

- 对 Claude Code Channels 路线聚焦
- 结构直接
- 适合做 channel 桥接演示和研究

它更像一个 **Channels 插件实现**。

本项目和它的区别在于：

- 本项目把 `channels` 降成了高级模式，而不是默认路径
- 本项目的主路径不依赖当前 Claude Code 会话在线
- 本项目在 API key / 自定义网关 / Channels 不可用的环境里，仍然能通过 worker 模式运行

这意味着本项目在真实使用场景里更稳：

- 对普通用户更友好
- 对本机后台派活更自然
- 对会话在线状态依赖更小

### 本项目的独特创新点

#### 1. 双模式产品架构

不是二选一，而是：

- `worker-first` 做产品主路径
- `channels` 做高级扩展层

这让产品和协议研究能共存，而不是互相拉扯。

#### 2. 会话与通道隔离

项目里专门做了两层隔离：

- worker / channels 独立 sync cursor
- worker 子进程与 MCP 工具隔离

这在真实稳定性上比很多演示型桥接项目更关键。

#### 3. 面向本地开发场景的服务化

它不是一个只在命令行演示里工作的桥接器，而是：

- 可后台运行
- 可开机启动
- 可状态查询
- 可停止和恢复

这让它更接近真正的本地 AI Agent 产品。

## 这件事对行业有什么借鉴意义

这个项目最大的借鉴意义，不是“把微信接上了 AI”，而是：

> 把移动端即时通信工具，变成了本地 AI 开发代理的轻量控制面板。

这有几个重要启发：

### 1. AI Agent 不一定要有新 App

很多人会本能地想到做一个新前端、新面板、新聊天界面。

但真正高频的入口，往往已经存在：

- 微信
- Slack
- Telegram
- Discord

如果把这些即时通信入口和本地 Agent 运行时桥起来，用户成本会低很多。

### 2. 产品主路径不一定等于最前沿协议

Channels 很先进，但不一定适合做默认产品路径。

产品决策应当优先考虑：

- 可安装
- 可解释
- 可维护
- 可后台运行

而不是优先考虑“最新最酷的协议能力”。

### 3. 先把单场景做深，比做大而全更值

本项目只盯住一个核心场景：

- 微信给本地 Claude Code 派活

正因为聚焦，才有机会把扫码、会话恢复、后台运行、状态回传、停止控制这些工程细节做扎实。

## 最后一句话

如果要用一句稍微专业、但大众也能看懂的话描述这个项目，可以这样说：

> 这是一个以 `worker-first` 为核心的本地 AI Agent 桥接产品。它把微信变成了 Claude Code 的远程任务入口；对用户来说，它是一个可安装、可后台运行的微信派活工具；对架构来说，它内部采用 SDK 式桥接抽象，并保留可选的 Channels 高级模式。
